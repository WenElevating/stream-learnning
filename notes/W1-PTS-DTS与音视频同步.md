## 📌 本节一句话总结

B 帧让"解码顺序≠显示顺序"，所以需要 DTS（解码时间）和 PTS（显示时间）两个时间戳；时间戳是"刻度"不是"秒"，要乘 time_base 换算；音视频两路时钟独立必然漂移，靠"音频主同步"策略让视频追音频。

## 1. ⭐ 编码顺序 vs 显示顺序

### 1.1 三种顺序，必须分清

| 顺序 | 含义 | 由谁决定 |
|---|---|---|
| **采集/显示顺序** | 按时间先后发生 | 摄像头采集顺序 = 观众看到的顺序 |
| **编码顺序（=存储顺序=DTS顺序）** | 文件里实际存的顺序 | 编码器重排后的顺序 |
| **解码顺序** | 解码器读入的顺序 | 等于编码顺序 |

> B 帧需要"后面的"I/P 帧做参考才能解码，所以编码器会把后面的 I/P 提前存进来。

### 1.2 用 GOP 看错位（bf=2，即每两个B帧之间一个P/I）

```
显示顺序(PTS排序):  I(0) B(1) B(2) P(3) B(4) B(5) P(6) ...
解码顺序(DTS排序):  I(0) P(3) B(1) B(2) P(6) B(4) B(5) ...

DTS 单调递增:       0    1    2    3    4    5    6
PTS 跳动:           0    3    1    2    6    4    5
```

### 1.3 解码缓冲区在干什么

```
读入 I(0) → 解码 → 缓冲（PTS=0，等显示时刻）
读入 P(3) → 解码 → 缓冲（PTS=3，要等前面显示完）
读入 B(1) → 解码 → PTS=1 正好到了 → 立即显示
读入 B(2) → 解码 → PTS=2 到了 → 显示
...（此时缓冲区里有 I(0) 和 P(3)，按 PTS 排序逐个显示）
```

> 💡 **核心结论**：B 帧迫使解码器必须缓存后续 I/P 帧才能解码当前 B 帧 → 产生"解码延迟"。
> **无人机图传要 `-bf 0` 关掉 B 帧，就是为了消除这个延迟。**

### 1.4 实测验证（w1_bf.ts，TS 封装）

```
ffprobe -v error -select_streams v -show_entries packet=pts,dts -of csv w1_bf.ts

前几包:
  PTS=132000  DTS=126000   (关键帧)
  PTS=141000  DTS=129000
  PTS=135000  DTS=132000   ← PTS 比上一包小! 错位铁证
  PTS=138000  DTS=135000
```

✅ DTS 单调递增，PTS 跳动 → 印证 B 帧重排。

## 2. ⭐ time_base 与时间戳换算

### 2.1 核心公式

```
实际时间(秒) = pts × time_base
```

pts 是"刻度"，time_base 是"每刻度多少秒"。

### 2.2 常见 time_base

| 封装 | time_base | 起始 PTS | 说明 |
|---|---|---|---|
| MP4 | `1/15360` | 通常 0 | 视频流（15360=512×30） |
| MP4 音频 | `1/44100` | 通常 0 | = 采样率 |
| MPEG-TS | `1/90000` | **可能非0** | 标准固定值（27MHz/300） |
| FLV | `1/1000` | 0 | 毫秒为单位 |

### 2.3 TS 为什么是 90000？

```
MPEG-TS 标准(ISO 13818-1): 系统时钟 27MHz / 300 = 90000 Hz
固定值，所有 TS 流都用它。

验算帧间隔:
  30fps → 每帧 90000/30 = 3000 刻度
  3000 × (1/90000) = 0.0333 秒 = 1/30 秒 ✓
```

### 2.4 ⚠️ 新手必踩的坑

1. **PTS 不一定从 0 开始**！TS 流常有起始偏移。→ 比较时间用"相对差值"，不要用绝对值。
2. **不同流有不同 time_base**！见 §4 实测，视频 15360 vs 音频 44100。
3. **跨流比较必须换算到同一 time_base**，用 `av_rescale_q()`。

### 2.5 FFmpeg 关键 API（W10 编程会用到）

```c
AVStream->time_base              // 这一路流的时间基
AVFrame->pts / AVPacket->pts     // 时间戳（time_base 单位）
av_rescale_q(pts, from_tb, to_tb) // 跨时间基换算 ★最常用
av_q2d(time_base)                // time_base 转成 double 秒数
```

## 3. ⭐ 音视频同步三大策略

### 3.1 为什么必然不同步

```
视频流：30fps → 每 33.3ms 一帧      独立时钟 A
音频流：48kHz  → 每帧 1024 采样≈21ms  独立时钟 B

两套时钟完全独立 → 没有同步机制必然漂移
硬件：摄像头晶体 ≠ 麦克风晶体（都有偏差）
网络：两路包走不同路径，延迟不同
```

### 3.2 三种同步策略

| 策略 | 主时钟 | 做法 | 用在哪 |
|---|---|---|---|
| ① 视频主同步 | 视频 | 音频跟视频 | 极少 |
| ② **音频主同步** ⭐ | 音频 | **视频跟音频** | **直播/图传主流** |
| ③ 外部时钟同步 | 系统墙钟 | 音视频都跟它 | 视频会议 |

### 3.3 为什么音频主同步是主流

```
人耳对声音断续极敏感（卡顿 0.1 秒就难受）
人眼对画面跳变较宽容（少一帧看不出）

→ 音频绝不变速，视频来调整
→ 直播卡顿常见"画面快进跳一下" = 视频在追音频
```

### 3.4 视频追音频的核心算法

```
每帧视频显示前，算与音频主时钟的差:
  diff = video_pts - audio_pts

  diff > 阈值      → 视频超前, 这一帧等一会 (sleep)
  diff < -阈值     → 视频落后, 丢帧或快进追上
  |diff| ≤ 阈值    → 正常显示
  (阈值一般 10~40ms, 人感知不到)
```

### 3.5 无人机图传特殊情况

- 图传常**只有视频没音频** → 退化为"系统墙钟为主时钟"，逻辑更简单
- 若有语音对讲 → 必须严格音频主同步

### 3.6 推流端常见同步坑（以后会逐一验证）

1. PTS 没设或设错 → 解码端按错顺序显示 → 音画不同步
2. 音视频 PTS 用了不同 time_base 没换算 → 时间戳对不上
3. 网络抖动导致包乱序 → 接收端要 jitter buffer 重排
4. B 帧本身的编码延迟 → 推流延迟 + 解码延迟 + 网络延迟 叠加

## 4. 实测验证

### 4.1 看 MP4 视频流的 time_base 和帧数

```
ffprobe -v error -select_streams v -show_entries stream=time_base,nb_frames,r_frame_rate w1_av.mp4
→ time_base=1/15360, nb_frames=90, r_frame_rate=30/1
验算: 90帧 / 30fps = 3.000 秒 ✓
```

### 4.2 看 MP4 音频流的 time_base 和帧数

```
ffprobe -v error -select_streams a -show_entries stream=time_base,nb_frames,sample_rate w1_av.mp4
→ time_base=1/44100, nb_frames=131, sample_rate=44100
```

### 4.3 ⭐ 关键洞察：视频和音频用了完全不同的 time_base

```
视频: 1/15360
音频: 1/44100    ← 两套独立时间系统!

→ 必须用 av_rescale_q() 跨流换算才能比较
→ 这就是为什么音视频会漂移、需要同步机制
```

## 5. 无人机场景串联

```
载荷端:
  摄像头 → ISP → [YUV420P, 视频时钟]
  麦克风 → [PCM, 音频时钟]              ← 两套时钟已独立
  
  编码器给每帧打 PTS（视频用自己的 tb，音频用自己的 tb）
  ↓
RTSP/SRT 打包传输（封装层用第三种 tb，如 TS 的 1/90000）
  ↓
地面端:
  解码 → 拿到原始 PTS
  同步模块: 以音频为主时钟，视频超前等、落后丢
  渲染显示
```

**你在推流代码里要确保**：每一帧的 PTS 都正确设置、音视频 time_base 一致或正确换算。这是音画同步的地基。

2. MPEG-TS 标准规定系统时钟 27MHz/300=90000Hz。30fps 每帧 90000/30=3000 刻度。
3. 不一定。TS 常有起始偏移。比较时间用"相对差值"。
