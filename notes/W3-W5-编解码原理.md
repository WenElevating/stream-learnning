## 📌 本节一句话总结

H.264 压缩 4000:1 的秘密 = 预测（I帧靠空间相关性、P/B帧靠时间冗余运动估计）+ 残差 + 熵编码；编码后输出的是 NALU（7=SPS/8=PPS/5=IDR/1=P或B）；无人机图传靠 `-bf 0 -g 30 -b:v CBR -tune zerolatency` 在带宽和延迟间求平衡。

# W3 · 预测机制与 GOP

## 3.1 编码三步走（地基）

```
① 空间预测 (Intra/帧内)   → I帧用
② 时间预测 (Inter/帧间)   → P/B帧用 ⭐压缩比最大
③ 变换+量化+熵编码         → 残差再压一遍
```

**核心公式**：`编码数据 = 预测参数 + 残差`。预测越准，残差越小，码率越低。

编码流程：
```
原始YUV → 预测 → 残差 = 原始 - 预测 → 变换+量化 → 熵编码 → NALU
```

---

## 3.2 帧内预测 Intra（I帧）

- I 帧不参考其他帧，靠**块内空间相关性**
- H.264 对 4×4 块提供 9 种预测方向（垂直/水平/对角/DC...）
- 编码器选预测最准的方向，只存"方向 + 残差"

### IDR 帧（特殊 I 帧）⭐

```
普通 I 帧：不参考其他帧，但后面的帧仍可参考它之前的帧
IDR 帧  ：后面的帧绝对不能参考它之前的任何帧 → 硬切分点
```

**IDR 的应用**：
- 切换码率（ABR）必须等 IDR 才能切，否则花屏
- 观众中途加入从 IDR 开始解码
- 丢包恢复靠 IDR 重新同步

---

## 3.3 ⭐ 帧间预测 Inter（P/B帧）+ 运动估计

### 核心思想：用"上一帧 + 物体怎么动"描述当前帧

```
对每个宏块:
  在参考帧里搜索最相似的 16×16 区域
  记录: 运动矢量 (dx,dy) + 残差

压缩比:
  存原始宏块(16×16 YUV420): 384字节
  存运动矢量+残差:          ~10字节
  → 压缩比 38:1
```

### P 帧与 B 帧

| | 参考方向 | 压缩比 | 延迟 |
|---|---|---|---|
| P 帧 | 只参考过去 | 中 | 低 |
| **B 帧** | 过去+未来 | **最高** | **高（需未来帧先编码）** |

### ⚠️ 无人机为什么关 B 帧

```
B 帧代价:
  - 编码延迟 +1帧 (等未来帧)
  - 解码延迟 +1帧 (缓冲)
  - 计算复杂 3-5×
  - 30fps下至少 66ms 额外延迟

无人机低延迟需求 → 必须 -bf 0
代价: 压缩比下降 20-30%
```

### 实测（w3_no_bf.mp4 vs w3_with_bf.mp4）

```
无B帧 (I/P):  109 kbps
有B帧 (I/P/B): 82 kbps   ← B帧省 25%

逐帧大小:
  I帧 ≈ 5000 字节
  P帧 ≈ 300 字节
  B帧 ≈ 50 字节    ← B帧只有I帧的 1%!

比例 I:P:B ≈ 100:10:1
```

---

## 3.4 ⭐ GOP 结构

```
GOP = 1个I帧 + 后续P/B帧, 直到下一个I帧
  I  B  B  P  B  B  P ... I  ...
  ├──── GOP ────────────┤

N = GOP长度（两个I帧距离）
M = 两个参考帧之间的B帧数+1
```

### GOP 长度核心权衡

| GOP | 抗丢包 | 压缩效率 | 切换速度 | 适用 |
|---|---|---|---|---|
| 1 (All-Intra) | ⭐极强 | 极差(8.5×码率) | 极快 | 极限低延迟FPV |
| 10 | 强 | 差(1.55×) | 快 | 弱网 |
| **30** (1秒) | 中 | 基准 | 中 | **直播/图传主流** ⭐ |
| 60 (2秒) | 弱 | 好(0.85×) | 慢 | 点播 |
| 300 (10秒) | 极弱 | 最好(0.71×) | 极慢 | 存储 |

### 实测（testsrc 10秒 640x360）

```
GOP=1:   945 kbps (8.5×)
GOP=10:  172 kbps (1.55×)
GOP=30:  111 kbps (基准)
GOP=60:   94 kbps (0.85×)
GOP=300:  79 kbps (0.71×)
```

> 工程经验：地面端能等 1 秒花屏恢复，不能等 10 秒。`-g 30` + `-bf 0` 是大部分图传标配。

# W4 · 码流结构（NALU + SPS/PPS）

## 4.1 ⭐ NALU 结构

```
编码器输出不是帧, 是 NALU:
┌──────────┬─────────────────┐
│ 1字节头   │ RBSP(载荷)       │
└──────────┴─────────────────┘

1字节头:
  forbidden(1) | nal_ref_idc(2) | nal_unit_type(5)
```

### nal_unit_type 速记

| Type | 名称 |
|---|---|
| 1 | 非IDR (P/B帧) |
| 5 | IDR (I帧) |
| 6 | SEI |
| **7** | **SPS** |
| **8** | **PPS** |

### 首字节速查表（看裸流第一眼）

```
0x67 = SPS     (0110 0111: ref=3 type=7)
0x68 = PPS     (0110 1000: ref=3 type=8)
0x65 = IDR I帧 (0110 0101: ref=3 type=5)
0x41 = P帧     (0100 0001: ref=2 type=1)
0x01 = B帧     (0000 0001: ref=0 type=1)
0x06 = SEI     (0000 0110: ref=0 type=6)
```

### 起始码

```
4字节: 00 00 00 01  (常用)
3字节: 00 00 01     (省1字节)

裸流结构:
  00 00 00 01 [SPS] 00 00 00 01 [PPS] 00 00 00 01 [IDR] ...
```

### 实测（w3_raw.h264）

```
共 65 个 NALU:
  序号0  SPS    30字节   0x67
  序号1  PPS    10字节   0x68
  序号2  SEI   613字节   0x06
  序号3  IDR  4408字节   0x65  ← 最大
  序号4+ P帧   ~300字节  0x41
```

---

## 4.2 ⭐ SPS / PPS

### SPS = Sequence Parameter Set（序列参数集）

整个视频序列的公共信息，解码器必须先解 SPS：

```
第2字节 profile_idc:
  66  → Baseline (最简, 无B帧)
  77  → Main
  100 → High ⭐ (主流, B帧+CABAC+8x8)

第4字节 level_idc (÷10 = Level):
  30 → Level 3.0 (720×576@25)
  40 → Level 4.0 (1080p@30) ⭐ 无人机1080p
  50 → Level 5.0 (4K@30)
```

### PPS = Picture Parameter Set（图像参数集）

每帧共用的图像级参数（10字节左右）：
- 熵编码方式（CAVLC/CABAC）
- 参考帧数量
- 初始量化参数

> ⭐ **推流关键**：SPS+PPS 必须在每个 IDR 前传给解码器。RTSP DESCRIBE / FLV视频Tag / WebRTC SDP 都携带它们。

---

## 4.3 ⭐ 码率控制

### 三种模式

| 模式 | 特点 | 适用 |
|---|---|---|
| **CBR** | 固定码率，网络友好 | **直播/图传首选** ⭐ |
| VBR | 可变码率，画质稳 | 文件存储 |
| CRF | 恒定质量 | 质量优先 |

### 无人机为什么用 CBR

```
带宽硬上限（4G/5G有限）
→ 必须锁定码率不超限
→ CBR: -b:v 4M -minrate 4M -maxrate 4M -bufsize 4M
代价: 复杂场景画质变差，但比丢包好
```

### 实测（mandelbrot 复杂源）

```
CRF 18 (高质量): 3421 kbps
CRF 28 (低质量): 1096 kbps
→ CRF 码率随复杂度变化, 不可控 → 不能用于带宽受限场景
```

# W5 · H.265 vs H.264 vs AV1

## 5.1 三代对比

| | H.264 | H.265 | AV1 |
|---|---|---|---|
| 发布 | 2003 | 2013 | 2018 |
| 压缩效率 | 基准 | 省40-50% | 再省20-30% |
| 同画质码率(1080p) | 4Mbps | **2Mbps** | 1.5Mbps |
| 编码复杂度 | 1× | 5-10× | 10-20× |
| 硬件支持 | 全平台 | 大部分新平台 | 较新 |
| 授权 | 有 | 贵 | **免费** |

### H.265 为何更高效

```
H.264: 固定 16×16 宏块
H.265: 可变 8×8 ~ 64×64 CTU
  → 天空用大块省码率, 细节用小块保质量
  + 35种帧内预测(vs H.264的9种)
```

### 实测（同 CRF=23, 5秒720p）

```
H.264: 114.7 kbps, 75015 字节
H.265: 72.0 kbps, 50756 字节  ← 省 32%
```

> 同样 4Mbps 带宽：H.265 能传 1080p，H.264 只能传 720p。

---

## 5.2 ⚠️ 无人机硬解坑

```
H.265 地面端兼容性问题:
  - 安卓 MediaCodec: 骁龙835前/低端芯片可能不稳
  - 浏览器 WebRTC: Chrome 2023后才支持
  - RTMP 协议不传 H.265 (需 Enhanced RTMP)

工程选择:
  嵌入式地面站 → H.265 硬解 OK
  移动 APP    → 看机型, H.264 兼容更广
  浏览器      → 仍推 H.264
  → 主流用 H.264, 高端选 H.265
```

---

## 5.3 无人机编码器选型决策树

```
Q1 地面端有专用硬编?
  → H.265 硬编 (NVENC/VAAPI/海思), 省功耗省带宽

Q2 要支持浏览器 WebRTC?
  → H.264 (兼容最好), 或双码流

Q3 极致低延迟 FPV?
  → H.264 + All-Intra (-g 1 -bf 0 + ultrafast)
     H.265 编码慢, 低延迟反不如 H.264

Q4 消费级无人机?
  → H.264 High profile 仍是稳妥选择
```

## 5.4 ⭐ 无人机图传标准编码参数（终极模板）

```bash
ffmpeg -i 摄像头 -c:v libx264 \
  -tune zerolatency \           # 零延迟调优
  -preset ultrafast \           # 编码最快 (压缩比换速度)
  -profile:v high \             # High profile
  -bf 0 \                       # 关B帧 (省2帧延迟)
  -g 30 \                       # 1秒一个I帧 (抗丢包1秒恢复)
  -b:v 4M -minrate 4M -maxrate 4M -bufsize 4M \  # 严格CBR
  -pix_fmt yuv420p
```

每个参数的来由，都对应本节某个原理：
- `-bf 0` ← §3.3（关 B 帧省延迟）
- `-g 30` ← §3.4（GOP 权衡）
- `-b:v/-minrate/-maxrate` ← §4.3（CBR 码率控制）
- `-tune zerolatency -preset ultrafast` ← 速度换延迟

<summary>🔑 参考答案</summary>

1. 空间预测(I)+时间预测(P/B)+变换量化熵编码。核心公式：`编码数据 = 预测参数 + 残差`，预测越准残差越小。
2. I:P:B ≈ 100:10:1。B帧最小因双向参考残差几乎为0（过去+未来帧平均），P帧次之（单向参考），I帧最大（存完整预测信息）。
3. B帧引入至少2帧延迟（编码等未来帧+解码缓冲），30fps下66ms，加上其他延迟破200ms飞手发飘。代价是压缩比降20-30%。
4. GOP=1抗丢包极强但码率暴涨8.5×；GOP=300压缩最好但丢包花屏10秒。无人机主流 `-g 30`（1秒恢复）。
5. NALU起始码。0x67=SPS, 0x65=IDR I帧, 0x41=P帧（ref=2）。
6. profile决定可用特性（High支持B帧+CABAC），level决定最大分辨率/码率（Level 4.0=1080p@30）。无人机1080p用 High@Level 4.0。
7. 解码器必须先解 SPS/PPS 才能解 IDR。没 SPS/PPS 解不开关键帧。
8. CBR固定码率网络友好（直播首选），VBR画质稳（存储），CRF恒定质量。无人机带宽硬上限必须用CBR锁定。
9. 省30-50%。因 H.265 用可变大小 CTU（8×8~64×64），简单区域大块省码率、复杂区域小块保细节，且帧内预测模式从9种增到35种。
10. 硬解兼容性。安卓低端芯片/浏览器WebRTC/RTMP协议都可能不支持或不稳。
11. 见笔记 §5.4 模板。每个参数对应：-bf 0←§3.3, -g 30←§3.4, CBR三参数←§4.3, zerolatency+ultrafast←速度换延迟。
