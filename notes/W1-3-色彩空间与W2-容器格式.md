# 第一篇：色彩空间与 HDR

## 1. 为什么需要色彩空间

视频里存的不是"颜色"而是"数字"，**同一个数字在不同设备上代表不同实际颜色**，必须靠"色彩空间"翻译规则才能正确还原。

常见症状：
- 手机鲜艳，电脑发灰 → 色彩范围不一致
- 推流后画面蒙雾 → Limited/Full 标错
- HDR 视频在 SDR 屏幕发暗 → 传输函数不匹配

---

## 2. ⭐ 色彩范围：Limited vs Full（推流发灰第一元凶）

### 2.1 两种取值范围

```
Full Range (PC/JPEG): 0~255 全用，0=纯黑 255=纯白
Limited Range (TV/MPEG): 只用 16~235 (Y) / 16~240 (UV)
   16=纯黑 235=纯白
   0~15 和 236~255 留给"过冲/欠冲"信号
   ⭐ 广播/电视/视频编码的默认标准
```

### 2.2 ⚠️ 推流发灰的坑（必遇）

```
载荷编码输出 Limited (Y:16~235)
地面端解码以为是 Full (Y:0~255)
→ Y=16 被当成深灰, Y=235 被当成浅灰
→ 整个画面发灰、对比度低、像蒙雾

反之: 载荷 Full, 地面端 Limited → 过曝、暗部糊
```

FFmpeg 标识：
- `AVCOL_RANGE_JPEG` = Full
- `AVCOL_RANGE_MPEG` = Limited

### 2.3 查看命令

```bash
ffprobe -v error -select_streams v -show_entries stream=color_range -of default=noprint_wrappers=1:nokey=1 视频.mp4
# 输出 limited / full / unknown(unknown 是大坑!)
```

### 2.4 实测验证（w1_color.mp4）

用 signalstats 看 Y 分量：
```
YMIN=6   (< 16, 过冲区域)
YMAX=245 (> 235, 过冲区域)
```
→ 印证 Limited Range 的"超白/超黑"区间。

## 3. ⭐ YUV↔RGB 转换矩阵

YUV 和 RGB 是同一颜色的两种坐标，转换靠矩阵。不同分辨率用不同矩阵：

| 标准 | 适用 | FFmpeg 标识 |
|---|---|---|
| BT.601 | SD 标清 | bt601 |
| **BT.709** ⭐ | HD 720p~4K | bt709 |
| BT.2020 | UHD 4K HDR | bt2020 |

> 无人机 1080p/720p 几乎都是 BT.709。

### 3.1 BT.709 转换公式（不用死记，见过即可）

```
Y = 0.2126·R + 0.7152·G + 0.0722·B    ← G 权重最大
R = Y + 1.5748·(V-128)
G = Y - 0.1873·(U-128) - 0.4681·(V-128)
B = Y + 1.8556·(U-128)
```

> ✅ **关键洞察**：Y 主要由 G（绿）决定（0.7152），人眼对绿色最敏感。这和 YUV420 色度降采样一脉相承——都是利用人眼特性省带宽。

### 3.2 矩阵用错的后果

载荷 BT.709 编码，地面端 BT.601 解码 → 红偏橙、绿偏黄、肤色异常。

FFmpeg 字段：`color_space` = bt601/bt709/bt2020

### 3.3 ⚠️ 实测踩坑

生成时指定 `-color_trc bt709 -color_primaries bt709`，但 ffprobe 显示 `unknown`——**x264 默认不写这些元数据进 MP4**。这是色彩问题的常见根源。

## 4. Gamma 与 HDR 概览

### 4.1 人眼对暗部更敏感（对数感知）

→ Gamma 编码：暗部放大、亮部压缩，8bit 也能描述丰富暗部细节。

### 4.2 SDR vs HDR

```
SDR: 8bit, Gamma 2.2, ~100 nits, BT.709
HDR: 10/12bit, PQ/HLG 传输函数, 可达 10000 nits, BT.2020
     标准: HDR10(通用) / Dolby Vision / HLG(广电/无人机)
```

无人机场景：当前图传多为 SDR+BT.709+8bit；高端载荷开始上 **HLG**（向下兼容 SDR）。

## 5. 色彩相关 ffprobe 命令汇总

```bash
# 一次性看所有色彩字段
ffprobe -v error -select_streams v -show_entries \
  stream=color_range,color_space,color_transfer,color_primaries,pix_fmt \
  -of default=noprint_wrappers=1 视频.mp4

# 看 Y 分量范围 (验证 Limited/Full)
ffmpeg -y -i 视频.mp4 -vf "signalstats,metadata=print:file=-" -f null - 2>&1 | grep YMIN
```

# 第二篇：容器格式内部结构

## 6. 三种容器的定位差异

| 容器 | 设计目标 | 关键特性 | 用途 |
|---|---|---|---|
| MP4/MOV | 文件存储 | 索引集中，需随机读 | 本地文件、VOD |
| **MPEG-TS** | **广播传输** | **固定188字节包，可中途加入** | 数字电视、卫星、SRT底层 |
| **FLV** | **流媒体传输** | **结构简单、tag顺序追加** | **RTMP 推流唯一选择** |

---

## 7. ⭐ MP4 box 结构

### 7.1 核心结构

```
[ftyp] 文件类型 (isom/iso2/avc1/mp41 等兼容标识)
[mdat] ⭐ 实际音视频数据 (占文件95%+)
[moov] ⭐⭐ 索引目录 (每帧偏移量、时间戳、SPS/PPS)
```

### 7.2 为什么 MP4 不能直播

```
moov 记录每帧在 mdat 里的绝对偏移量
→ 必须先写完所有帧才能算出 moov
→ 默认 moov 在文件末尾
→ 直播流永远不结束 → 永远拿不到 moov → 解不开
```

### 7.3 实测（w1_color.mp4, 71612 字节）

```
ftyp offset=0     size=32      (isom/iso2/avc1/mp41)
free  offset=32    size=8
mdat  offset=40    size=69661   ← 97% 是数据
moov  offset=69701 size=1911    ← 末尾!
```

### 7.4 faststart

`-movflags +faststart` 把 moov 移到开头 → 点播加速。但本质还是 VOD，不能直播。

---

## 8. ⭐ MPEG-TS 包结构

### 8.1 包格式

```
每个 packet 固定 188 字节:
┌──────────┬─────────────────────────────┐
│ 4字节头部 │ 184字节载荷(可能含适配字段)    │
└──────────┴─────────────────────────────┘

4字节头部 bit 分布:
  同步字节(0x47) | TEI PUSI PRI | PID高 | PID低 | SC AP PP CC

  TEI  传输错误指示 (1=包损坏)
  PUSI 载荷单元起始 (1=新帧开始) ⭐
  PID  流标识 (PAT=0, PMT, 视频流, 音频流...)
  AP   适配字段存在 (含 PCR 时钟)
  CC   连续计数器 0~15 循环 (检测丢包) ⭐
```

### 8.2 为什么 188 字节

早期卫星用 RS(204,188) 纠错码：188+16=204，固化下来。

### 8.3 TS 解码流程

```
1. 读 PAT (PID=0) → 找到 PMT 的 PID
2. 读 PMT → 找到视频流/音频流的 PID
3. 按目标 PID 过滤包 → 重组 PES → 解码
```

### 8.4 与 MP4 对比

```
MP4: 索引集中(moov), 头坏全完, 不可中途加入
TS : 无集中索引, 拆188小包, 单包坏只丢一帧, ⭐ 可随时加入
```

### 8.5 实测（w1_bf.ts, 46060 字节 = 245 个包）

```
packet[0]: PID=0x0011(17)   PMT
packet[1]: PID=0x0000(0)    PAT
packet[2]: PID=0x1000(4096)
packet[3]: PID=0x0100(256)  视频流
```

> 无人机场景：SRT 协议本质 = TS 包 + UDP + ARQ 重传。

---

## 9. ⭐ FLV tag 结构

### 9.1 结构

```
FLV Header (9字节): "FLV" + 版本 + 音视频标志
PreviousTagSize0 (4字节)
Tag1 (11字节头 + 载荷)
PreviousTagSize1
Tag2
...

Tag 头 (11字节):
  TagType (1字节): 8=音频 9=视频 18=脚本
  DataSize (3字节): 载荷大小
  Timestamp (3+1字节): ⭐ 4字节毫秒时间戳
  StreamID (3字节): 固定0
Body: 实际音视频数据
```

### 9.2 为什么 FLV 适合 RTMP

```
1. 时间戳毫秒整数 → 简单直观
2. Tag 顺序追加 → 边推边写, 不需索引
3. 结构极简 → 解析快, 延迟低
4. Adobe 时代事实标准 → 服务器全支持
```

### 9.3 FLV 局限

- 不原生支持 H.265/AV1（需 Enhanced RTMP 扩展）
- 不支持 HDR 元数据
- 时间戳 ~49 天溢出（32位毫秒）

### 9.4 实测（w1.flv, 71834 字节）

```
Header: FLV v1 仅视频
Tag[0]: 脚本 262字节 0ms     ← onMetaData 元信息
Tag[1]: 视频 56字节  0ms     ← SPS/PPS 序列头
Tag[2]: 视频 12221字节 0ms   ← 关键帧(I帧)
```

---

## 10. 三容器终极对比

| 维度 | MP4 | MPEG-TS | FLV |
|---|---|---|---|
| 设计目标 | 文件存储 | 广播传输 | 流媒体传输 |
| 基本单元 | box（不定长） | **packet（固定188）** | tag（不定长） |
| 索引 | moov atom | 无 | 无 |
| 时间戳单位 | 1/15360 等 | **1/90000** | **1/1000（ms）** |
| 可直播 | ❌ | ✅ | ✅ |
| 中途加入 | ❌ | ✅ | ✅ |
| 抗丢包 | 差 | **强** | 中 |
| H.265 | ✅ | ✅ | ❌（需扩展） |
| 无人机用途 | 录像回传 | **SRT/卫星** | **RTMP推流** |

## 11. 无人机场景串联

```
载荷端编码 → 输出 [YUV420P + color_range=tv + bt709]
       ↓
封装选择:
  - 实时回传: FLV (走 RTMP) 或 TS (走 SRT)
  - 录像存储: MP4 (加 faststart 便于后续点播)
       ↓
传输 (W6-W8 详讲)
       ↓
地面端解码:
  - 必须正确读取 color_range / color_space
  - 否则画面发灰/偏色
```

<summary>🔑 参考答案</summary>

1. color_range 不匹配。载荷 Limited（16-235）地面端按 Full（0-255）解 → 发灰。排查：ffprobe 看 color_range，或用 signalstats 看 Y 范围。

3. 人眼对绿色最敏感（G 权重 0.7152）。和 YUV420 把色度降采样一脉相承——都利用人眼对亮度/绿色敏感、对色彩不敏感的特性省带宽。

6. 毫秒（1/1000）。因为 tag 顺序追加不需索引、结构简单解析快、毫秒时间戳直观、Adobe 时代服务器全支持。
7. TS。因为 TS 固定包长、抗丢包、可中途加入，适合不可靠网络。SRT = TS + UDP + ARQ 重传。
