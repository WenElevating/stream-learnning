## 📌 本节一句话总结

一段视频 = **容器**（箱子）里装着**流**（视频/音频），流是用**编码格式**（压缩袋）压缩过的；摄像头出 YUV 不出 RGB，是为了省一半带宽。

---

## 1. 三层结构（最易混淆，必须分清）

```
┌─────────────────────────────────────┐
│  容器 Container  (.mp4 .flv .ts .mkv)│  ← 快递箱
│  ┌──────────┬──────────┐            │
│  │ 视频流    │ 音频流    │ ← Stream   │  ← 箱子里的物品
│  │ H.264    │ AAC      │ ← 编码      │  ← 物品的压缩袋
│  └──────────┴──────────┘            │
└─────────────────────────────────────┘
```

| 概念 | 例子 | 作用 |
|---|---|---|
| **容器** | MP4、FLV、MKV、MPEG-TS | 把多路流打包成一个文件，含时间信息 |
| **流** | 视频流、音频流 | 一路独立媒体数据 |
| **编码格式** | H.264、H.265、AAC | 把原始数据压缩成更小的字节 |

> ⚠️ 常见误区：`.mp4` 不是"视频格式"，它只是箱子。同样一个 .mp4，里面装的可能 H.264 也可能 H.265。

---

## 2. 四个基本量

| 概念 | 定义 | 无人机场景 |
|---|---|---|
| 分辨率 | 横×纵像素数 | 1080p = 1920×1080 |
| 帧率 FPS | 每秒画面数 | 图传常 30 / 60 fps |
| 码率 Bitrate | 每秒数据量 | 4Mbps = 每秒 0.5MB |
| 像素格式 | 每个像素怎么存 | YUV420P |

### 🔢 必背公式

```
未压缩视频体积 = 分辨率 × 每像素字节 × 帧率 × 时长

1080p@30fps 一秒未压缩（YUV420，1.5 字节/像素）：
1920 × 1080 × 1.5 × 30 ≈ 93 MB/s

如果是 RGB（3 字节/像素）：≈ 186 MB/s
```

→ 这就是**为什么必须压缩**。无人机带宽只有几 Mbps，不压根本传不动。

## 3. ⭐ YUV vs RGB（W1 重点）

### 3.1 YUV 把像素拆成两部分

```
Y  = 亮度 Luma    ← 黑白电视时代就有，相当于灰度
U,V = 色度 Chroma  ← 给灰度"上色"
```

### 3.2 为什么用 YUV？——人眼特性

```
人眼对 亮度细节 敏感（轮廓、纹理）
人眼对 色彩细节 不敏感（看不出微小色差）

→ Y 全分辨率保留
→ U,V 降低分辨率也看不出来 → 省带宽
```

### 3.3 采样格式 J:a:b

| 格式 | 色度处理 | 每像素字节 | 用在哪 |
|---|---|---|---|
| YUV444 | 不降采样 | 3 | 高端后期 |
| YUV422 | 水平减半 | 2 | 广播 |
| **YUV420** | 水平+垂直都减半 | **1.5** | ⭐ 几乎所有视频编码 |

### 3.4 YUV420 视觉化（4×4 像素块）

```
Y:  4×4 = 16 个（全保留）    16 字节
U:  2×2 =  4 个（横竖各减半） +4 字节
V:  2×2 =  4 个              +4 字节
─────────────────────────
合计 24 字节 / 16 像素 = 1.5 字节/像素
比 RGB(3) 省一半
```

> ✅ 结论：摄像头 ISP 输出的原始帧几乎都是 **YUV420P**（P=planar，三分量分别连续存储）。FFmpeg 里记作 `AV_PIX_FMT_YUV420P`。
>
> 无人机场景：载荷 ISP → YUV420P → 编码器，整条链路默认 YUV，几乎不碰 RGB。

## 4. PTS / DTS（时间戳，W1 难点）

### 4.1 两个时间戳

```
DTS (Decoding Timestamp)  = 什么时候解码这一帧
PTS (Presentation Timestamp) = 什么时候显示这一帧

规则：DTS ≤ PTS（先解码才能显示）
```

### 4.2 为什么需要两个？——因为有 B 帧

```
文件里存的顺序（解码顺序 = DTS）:
  I  P  B  B  P
  1  2  3  4  5

实际播放顺序（显示顺序 = PTS）可能是:
  I  B  B  P  P
  1  3  4  2  5
```

B 帧（双向预测帧，W3 详讲）要参考"后面的帧"才能解码，所以存储顺序 ≠ 播放顺序。

### 4.3 为什么音视频会不同步？

- 视频按帧率（30fps → 每 33ms 一帧）
- 音频按采样率（48kHz → 每秒 48000 采样）
- 两路时钟独立 → 没同步机制就会漂移

> 同步方案（W2 详讲）：以"主时钟"为基准（通常音频），视频来早了等、来晚了丢。

## 5. 实测验证（W1-Lab1 已跑通）

用 `testsrc` 生成的 640×360/30fps/3秒/YUV420P 测试视频：

| 字段 | 值 | 说明 |
|---|---|---|
| codec_name | h264 | 编码 H.264 |
| pix_fmt | yuv420p | ⭐ 像素格式 |
| width×height | 640×360 | 分辨率 |
| r_frame_rate | 30/1 | 帧率 |
| nb_frames | 90 | 3秒×30fps=90 ✓ |
| bit_rate (流) | 56160 bps | 流码率 |
| bit_rate (容器) | 61373 bps | 含封装开销 |
| has_b_frames | 2 | 有 B 帧 → 需 DTS/PTS |

### 三组验证
1. **码率两值**：流 56160 < 容器 61373，差值≈封装开销
2. **压缩比**：未压缩 93.3MB → 编码后 23KB ≈ **4000:1**
3. **B帧**：has_b_frames=2 → 解码顺序≠显示顺序 → 需要 DTS+PTS

## 6. 无人机场景串联

把今天学的串到无人机图传链路：

```
摄像头 → ISP → [YUV420P 原始帧]  ← 今天讲的像素格式
              ↓
         H.264/H.265 编码器      ← W3-W5 讲编码原理
              ↓
         [压缩后的 NALU]          ← W4 讲码流结构
              ↓
         RTSP/SRT/WebRTC 封装     ← W6-W8 讲协议
              ↓
         4G/5G 弱网传输           ← W14 讲弱网对抗
              ↓
         地面端解码 → 渲染
```

今天的 YUV420P 就是这条链路的**起点数据格式**，后面所有环节都建立在这之上。
